# 데이터 분석 학습 회고

이 저장소는 완성된 프로젝트를 소개하기 위한 결과물이 아니라, NumPy와 통계 공부를 시작하면서 내가 무엇을 이해했고 어디에서 막혔는지를 남긴 학습 기록이다.

처음에는 배열을 만드는 것부터 시작했다. 이후 인덱싱과 슬라이싱, 팬시 인덱싱, 브로드캐스팅, 행렬곱, `axis` 연산으로 범위를 넓혔고 마지막에는 통계의 기본 개념까지 정리했다. 커밋 기록을 다시 살펴보니 공부한 순서와 이해가 확장된 과정이 그대로 남아 있었다.

## 작업 흐름을 돌아보며

### 1. ndarray를 처음 다뤘을 때

첫 단계에서는 Python의 리스트와 NumPy 배열이 어떻게 다른지 알아보는 데 집중했다.

- `np.array()`로 1차원부터 3차원 배열 만들기
- `shape`, `ndim`, `size`, `dtype`으로 배열의 구조 확인하기
- `zeros`, `ones`, `full`로 원하는 형태의 배열 만들기
- `arange`, `linspace`, `logspace`로 일정한 규칙의 값 생성하기

이때는 배열의 출력값보다 “몇 차원인지”, “원소가 몇 개인지”, “어떤 dtype으로 저장되는지”를 코드로 직접 확인하는 과정이 중요했다. 과일별 분기 판매량을 더하고 평균을 구한 실습을 통해 NumPy 연산이 반복문 없이도 데이터 전체에 적용된다는 점도 확인했다.

관련 기록: [`01_ndarray.ipynb`](01_numpy/01_ndarray.ipynb)

### 2. 값을 선택하는 방법을 익혔을 때

배열 생성 다음에는 필요한 데이터를 선택하는 방법을 공부했다.

처음에는 한 개의 값을 가져오는 기본 인덱싱과 범위를 가져오는 슬라이싱을 연습했다. 이후 인덱스 목록을 사용하는 팬시 인덱싱과 조건식으로 값을 거르는 불리언 인덱싱까지 확장했다.

특히 2차원과 3차원 배열은 처음부터 공간으로 이해하려고 하면 혼란스러웠다. 바깥쪽 축부터 차례대로 참조한다는 방식으로 접근하니 `arr[축, 행, 열]` 형태를 조금 더 명확하게 읽을 수 있었다.

관련 기록: [`02_indexing_slicing.ipynb`](01_numpy/02_indexing_slicing.ipynb)

### 3. 배열 연산에서 shape의 중요성을 배웠을 때

배열 연산을 공부하면서 값 자체만큼 shape가 중요하다는 것을 알게 됐다.

- 같은 위치의 원소끼리 계산하는 원소별 연산
- 서로 다른 shape를 맞춰 계산하는 브로드캐스팅
- 스칼라와 배열의 연산
- `*`, `np.multiply()`와 `@`, `np.dot()`의 차이
- `axis=None`, `axis=0`, `axis=1`에 따른 집계 결과

브로드캐스팅은 단순히 “행이나 열의 길이가 맞으면 된다”라고 기억하기보다 뒤쪽 차원부터 비교하고, 각 차원이 같거나 둘 중 하나가 1이어야 한다는 규칙으로 이해해야 했다.

행렬곱도 처음에는 원소별 곱과 혼동하기 쉬웠다. `(m, n) @ (n, p)`의 안쪽 크기가 같아야 하고 결과 shape는 `(m, p)`가 된다는 기준을 코드와 계산식으로 함께 확인했다.

`axis`는 아직도 주의해서 읽어야 하는 부분이다. 단순히 행과 열의 방향을 외우는 것보다 “연산 후 사라지는 축”이라고 이해하는 편이 결과를 예측하는 데 도움이 됐다.

관련 기록: [`03_operation.ipynb`](01_numpy/03_operation.ipynb)

### 4. 계산에서 통계의 의미로 넘어갔을 때

마지막에는 NumPy 함수를 사용하는 것에서 한 걸음 더 나아가, 계산한 숫자가 어떤 의미인지 정리했다.

- 모집단과 표본
- 기술통계와 추론통계
- 평균, 중앙값, 최빈값
- 범위, 분산, 표준편차
- 모집단 분산과 표본 분산
- 정규분포와 균등분포

평균 하나만으로 데이터를 설명하기 어렵고, 같은 평균을 가진 데이터도 퍼진 정도가 다를 수 있다는 점이 핵심이었다. 분산은 퍼짐을 수치화하지만 단위가 제곱되고, 표준편차는 다시 원래 단위로 돌아와 해석하기 쉽다는 흐름으로 이해했다.

`ddof=0`과 `ddof=1`의 차이도 단순한 옵션이 아니라 모집단과 표본을 구분하는 계산 기준이라는 점을 정리했다.

관련 기록: [`04_statistics.ipynb`](01_numpy/04_statistics.ipynb)

## 내 질문 방식과 GPT 활용에 대한 회고

이번 작업에서 나는 긴 요구사항을 먼저 작성하기보다 다음 행동을 짧게 요청하는 방식을 사용했다.

- “워닝을 확인해볼래?”
- 오류 메시지를 전달하고 “이거 왜 뜬 거야?”라고 원인을 확인하기
- Git 커밋과 코드, 주석을 함께 살펴보고 문서 작성을 요청하기
- 결과가 의도와 다르면 “프로젝트 설명이 아니라 회고록 느낌”이라고 방향을 다시 지정하기

내 질문은 짧고 직접적이며, 세부 절차보다는 원하는 결과를 먼저 제시하는 편이었다. 문장 끝의 “해볼래?”는 명령만 전달하기보다 GPT가 현재 상태를 직접 조사하고 판단하도록 맡기는 협업형 표현에 가까웠다.

이 방식은 빠르게 작업을 시작하기에는 효과적이었다. 반면 처음부터 문서의 독자, 목적, 말투를 충분히 지정하지 않으면 일반적인 프로젝트 소개문처럼 다른 결과가 나올 수 있다는 점도 확인했다. 이번 README가 한 번 수정된 이유도 “README 작성”만으로는 사용 안내서인지 학습 회고인지 목적이 분명하지 않았기 때문이다.

GPT는 다음과 같은 역할로 활용했다.

- 저장된 코드만 보는 것이 아니라 Git 커밋 순서와 원격 반영 상태까지 함께 확인
- 노트북의 마크다운, 코드, 주석에서 실제 학습 범위 추출
- 실행 환경 경고와 프로젝트 코드의 문제를 구분
- 내가 사용한 질문 표현과 수정 요청에서 원하는 문서의 관점 파악

결국 GPT를 잘 활용하려면 단순히 작업을 시키는 것뿐 아니라, 결과를 보고 의도와 다른 지점을 구체적으로 다시 전달하는 과정이 필요했다.

## 아쉬웠던 점

노트북에는 공부 당시의 생각이 주석으로 바로 기록되어 있어 학습 흔적은 잘 보이지만, 다음과 같은 부분은 보완이 필요하다.

- `axis`, 변수명, 출력 설명에 일부 오타가 남아 있다.
- 사용하지 않는 import와 빈 코드 셀이 있다.
- 일부 셀은 앞에서 선언한 변수에 의존하므로 단독 실행이 어렵다.
- 브로드캐스팅이 실패하는 예제는 주석 처리되어 있어 실제 오류 메시지를 확인하기 어렵다.
- 통계 노트북은 아직 개념 설명 중심이며 계산 및 시각화 실습이 부족하다.
- 노트북마다 저장된 Python 메타데이터가 일관되지 않다.

이것들은 단순히 지워야 할 실수라기보다 당시 어떤 부분을 헷갈렸는지 보여주는 기록이기도 하다. 다만 학습 자료로 다시 사용하려면 잘못된 설명과 오타는 바로잡고, 의도적으로 남긴 실패 예제에는 이유를 명확히 적어야 한다.
